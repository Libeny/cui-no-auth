# CUI 卡顿修复说明

## 这次修的是什么问题

这次主要修三个会影响 CUI 使用稳定性的地方。

第一个是 Codex 历史会话很多的时候，CUI 需要去 `~/.codex/sessions` 下面找很多 `rollout-*.jsonl` 文件。旧逻辑更像是每次从一堆文件里重新翻找和读取，文件越多，首次打开和刷新列表就越慢。

第二个是输入框的文件补全。你们现在主要用 CUI 看历史和监控任务，不太需要在输入框里 `@文件名`。旧逻辑会在输入框里做文件补全，如果当前目录很大，就可能递归扫描很多文件，拖慢页面。

第三个是默认模式。你们希望 CUI 默认是只读的，也就是打开页面后主要看会话，不要因为误操作启动新的 agent 或继续输入。

## 修复后的效果

修完后，CUI 不需要每次都完整重读所有 Codex 会话文件。它会把已经读过的会话摘要、时间、模型、文件位置等信息保存到本地 SQLite 索引里。下次打开列表时，优先直接读这个索引，所以会快很多，也更省内存。

如果某个 Codex JSONL 文件没有变化，CUI 会通过文件大小和修改时间判断它没变，然后跳过解析。只有新文件或变过的文件才会重新读取。

输入框文件补全已经在 Home 和会话页默认关闭。打开 CUI 时不会因为输入框补全去扫项目目录。

CUI 现在默认是只读模式。页面默认隐藏输入框，后端也会拒绝启动、继续、重命名、归档这类写操作。也就是说，不只是前端不显示输入框，就算有人直接调用启动接口，后端也会拦住。

底层文件列表接口仍然加了保护：即使以后某个页面还要用文件列表，也不会无限递归扫描目录，会限制最多返回数量和目录深度。

## 我用了什么技术

可以把这次改动理解成三件事。

1. **本地索引**

   CUI 已经有一个本地 SQLite 数据库。这次让 Codex 会话也使用这个数据库保存索引。SQLite 可以理解成一个轻量本地表格，不需要单独启动数据库服务。

2. **增量扫描**

   每个历史文件都有两个重要信息：文件大小和最后修改时间。如果这两个都没变，就说明文件内容基本没变，CUI 不再重复读它。

3. **只读保护**

   配置里新增了 `server.readOnly`，默认值是 `true`。服务启动后会把只读状态告诉前端，前端据此隐藏输入框；后端会在关键写接口上做二次拦截，避免绕过页面直接启动任务。

4. **受限扫描**

   文件列表、Claude Code 扫描、Codex 扫描都不能无限制地跑。文件列表限制目录深度和最多返回数量；Claude Code 默认 4 个扫描 worker，Codex 默认 6 个扫描 worker。两个索引器一起跑时，总扫描并发默认是 10，既比纯串行快，也不会一次性占太多内存。

## 对你使用上的变化

你们日常不需要改使用方式。照常打开 CUI、看 Codex/Claude 历史、点会话详情。

可见变化是：默认打开后没有输入框，也不会做 `@文件` 补全。这个符合你们“默认 read-only”的使用方式。

如果后面确实需要临时恢复输入和启动任务，可以用 `--allow-write` 启动服务；如果想显式保持只读，可以用 `--read-only`。

## 这次没有修的范围

这次重点是“很多 session 时也要快”“默认只读”“输入框不再做文件补全”。它没有改 Discord 附件传入传出能力。附件能力可以作为下一步单独做。
